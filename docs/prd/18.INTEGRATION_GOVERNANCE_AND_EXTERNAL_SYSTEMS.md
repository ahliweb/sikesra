# Integration Governance and External Systems — SIKESRA

## 1. Prinsip Integrasi

### 1.1 Tidak Ada Integrasi Eksternal Wajib di MVP

SIKESRA MVP (Sprint 0-4) **berdiri sendiri** — tidak memiliki dependensi keras ke sistem eksternal apa pun. Semua data wilayah, master data, dan dokumen dikelola di dalam plugin sendiri.

### 1.2 Adapter Pattern untuk Integrasi Masa Depan

Jika integrasi eksternal ditambahkan (Phase 2+), gunakan pola adapter: definisikan interface internal (mis. `RegionDataProvider`) yang diimplementasikan oleh sumber data lokal (default MVP) dan dapat diganti implementasi sumber eksternal tanpa mengubah kode pemanggil.

### 1.3 Graceful Degradation

Setiap integrasi eksternal di masa depan harus memiliki fallback — jika sistem eksternal tidak tersedia, SIKESRA tetap berfungsi dengan data lokal/cache terakhir, tidak boleh membuat seluruh modul gagal total.

### 1.4 Audit Semua Panggilan Eksternal

Setiap panggilan ke sistem eksternal (saat diimplementasikan) wajib menghasilkan audit event (`kind` berprefiks `integration.*`) — sukses maupun gagal.

## 2. Arsitektur Integrasi (Pola Umum, Belum Diimplementasikan)

```text
SIKESRA Plugin → Adapter Interface → [Implementasi Lokal (default) | Implementasi Eksternal (Phase 2+)]
```

## 3. Status Integrasi per Sistem (Saat Ini)

| Sistem | Status MVP | Catatan |
| --- | --- | --- |
| Data wilayah resmi Kemendagri | Tidak terintegrasi — input/seed manual | Kandidat Phase 2, lihat `21.ADDENDUM_MASTER_DATA_WILAYAH.md` |
| Sistem informasi kesejahteraan sosial nasional | Tidak terintegrasi | Belum ada keputusan, perlu Discussion jika dibutuhkan |
| Layanan notifikasi (email/WhatsApp) | Tidak terintegrasi — in-app saja di MVP | Kandidat Phase 2 |

## 4. Spesifikasi Integrasi Phase 2 (Kerangka Awal, Belum Final)

### 4.1 Sinkronisasi Data Wilayah Resmi

Jika diimplementasikan: adapter membaca data wilayah resmi terbaru secara periodik (cron), memvalidasi terhadap struktur hierarki yang sudah ada, dan tidak menimpa data entitas registry yang sudah ada — hanya memperbarui tabel referensi wilayah.

### 4.2 Notifikasi Phase 2

Jika diimplementasikan: adapter notifikasi terpisah dari logika verifikasi inti — kegagalan kirim notifikasi tidak boleh menggagalkan transisi stage verifikasi itu sendiri (lihat prinsip 1.3 Graceful Degradation).

## 5. Service Level Agreements (SLA) per Integrasi

Belum relevan untuk MVP karena tidak ada integrasi aktif. Setiap integrasi baru di masa depan wajib mendefinisikan SLA (timeout, retry policy, circuit breaker) sebagai bagian dari Change Request (`12.CHANGE_CONTROL_AND_DECISION_LOG.md` § 5) sebelum diimplementasikan.

## 6. Keamanan Integrasi (Prinsip untuk Masa Depan)

### 6.1 API Key Management

Kredensial integrasi eksternal disimpan sebagai Cloudflare secrets, mengikuti pola `import.meta.env.EMDASH_X || import.meta.env.X || ""` — tidak pernah di-hardcode atau di-commit.

### 6.2 Audit Outbound Calls

Setiap panggilan keluar wajib tercatat di `auditEvents (collection)` dengan `kind: "integration.call"`, termasuk status sukses/gagal (tanpa mencatat payload sensitif penuh di `metadata`).

### 6.3 Minimisasi Data

Integrasi keluar hanya mengirim field yang benar-benar diperlukan sistem tujuan — tidak mengirim seluruh baris registry entity secara default.

## 7. Testing Integrasi (Prinsip untuk Masa Depan)

### 7.1 Mock/Stub untuk Pengembangan

Implementasi adapter eksternal harus dapat diuji dengan mock/stub lokal tanpa memerlukan koneksi ke sistem eksternal sungguhan saat menjalankan test suite (`pnpm test`).

### 7.2 Kriteria Go-Live per Integrasi

Integrasi baru tidak boleh masuk produksi sebelum: (1) adapter teruji dengan mock, (2) graceful degradation teruji (sistem eksternal down → SIKESRA tetap berjalan), (3) audit event integrasi terverifikasi tercatat.
