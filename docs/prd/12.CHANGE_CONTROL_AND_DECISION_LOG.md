# Change Control and Decision Log — SIKESRA

## 1. Tujuan Dokumen

Mencatat keputusan desain yang sudah diambil (agar tidak didebat ulang tanpa alasan baru) dan mendefinisikan proses pengajuan perubahan scope/arsitektur setelah PRD disetujui.

## 2. Proses Change Control

### 2.1 Kapan Change Control Diperlukan

- Mengubah invariant di `01.AI_IMPLEMENTATION_PROMPT.md` (HR-01 s.d. HR-08).
- Menambah/mengubah stage verifikasi atau urutan level pengguna.
- Mengubah schema collection storage yang sudah dirilis ke produksi (breaking change pada shape data yang tersimpan di D1).
- H2-01 (rekonsiliasi `VerificationLevel` vs `VerificationUserLevel`) — mengubah union type yang dipakai di route `verification/advance`/`reject` adalah perubahan breaking pada perilaku route yang sudah ada.
- Menambah dependensi/integrasi eksternal baru.

Perubahan kecil (menambah field opsional, menambah permission baru, menambah halaman admin) **tidak** memerlukan change control formal — cukup PR + changeset seperti biasa.

### 2.2 Alur Change Control

1. Buka GitHub Discussion di kategori "Ideas" yang menjelaskan perubahan dan alasannya.
2. Jika disetujui, perbarui dokumen PRD terkait pada PR yang sama dengan implementasi, atau PR dokumentasi terpisah sebelum implementasi.
3. Catat keputusan di bagian § 3 dokumen ini dengan nomor `DEC-NNN`.

### 2.3 SLA Penanganan Change Request

Tidak ada SLA waktu tetap untuk MVP (tim kecil) — perubahan didiskusikan secara asinkron di Discussion sebelum PR dibuka.

## 3. Log Keputusan Desain

### DEC-001: Plugin Native, Bukan Standard/Sandboxed sebagai Default

**Keputusan**: Gunakan format native plugin (`src/index.ts` + `definePlugin()`) sebagai jalur utama instalasi; sandboxed entry (`src/sandbox.ts`) disiapkan untuk kebutuhan marketplace di masa depan, bukan jalur utama MVP.

**Alasan**: Native plugin memberi akses penuh ke EmDash internals yang dibutuhkan untuk hook lifecycle dan manifest navigasi tanpa overhead sandboxing yang belum diperlukan di MVP.

### DEC-002: Storage Plugin-Owned, Tidak Menyentuh Tabel Core

**Keputusan**: Semua data SIKESRA disimpan lewat `PluginStorageConfig` collections milik plugin sendiri (`ctx.storage.<collectionName>`), dibuat/di-seed di hook `plugin:install`/`plugin:activate`, dan tidak pernah memodifikasi tabel `_emdash_*`.

**Alasan**: Menjaga upgradability — sinkronisasi upstream EmDash tidak boleh berisiko merusak data SIKESRA, dan sebaliknya.

**Catatan koreksi (Juni 2026)**: Keputusan ini awalnya didokumentasikan secara salah sebagai "tabel SQL berprefiks `sikesra_*`" — audit kode menemukan plugin tidak pernah menulis SQL, lihat `03.PLUGIN_ARCHITECTURE.md` § 3 dan § 8.

### DEC-003: RBAC + ABAC sebagai Dua Lapis Terpisah

**Keputusan**: RBAC menjawab "boleh melakukan aksi apa", ABAC menjawab "data mana yang boleh diakses". Keduanya dicek di setiap route state-changing, bukan saling menggantikan.

> **Catatan implementasi (Juni 2026)**: Ini adalah keputusan TARGET. Kondisi saat ini: tidak ada route mutasi yang mengecek RBAC atau ABAC sebagai gerbang nyata. Backlog H1-02 dan H1-03 adalah pekerjaan untuk mewujudkan keputusan ini.

**Alasan**: Domain SIKESRA punya dimensi wilayah (desa/kecamatan/SOPD/kabupaten) yang tidak bisa direpresentasikan murni dengan role — petugas desa A dan petugas desa B punya role yang sama tapi scope data berbeda.

### DEC-004: 9 Stage Verifikasi Linear, Tidak Bisa Lompat

**Keputusan**: Stage verifikasi (`draft` → ... → `active_verified`) hanya dapat naik satu tingkat per transisi melalui endpoint resmi.

**Alasan**: Menjamin jejak audit lengkap di setiap level pemerintahan — melompati level menghilangkan akuntabilitas verifikasi di level tersebut.

### DEC-005: Default-Deny pada ABAC Engine

**Keputusan**: Jika tidak ada policy ABAC yang cocok untuk kombinasi subject/resource/action, hasil evaluasi adalah `deny`.

**Alasan**: Data dalam SIKESRA mencakup kategori sensitif (disabilitas, anak yatim, lansia terlantar) — fail-safe harus menutup akses, bukan membuka.

### DEC-006: Sensitivity Level Terpisah dari Verification Stage

**Keputusan**: `sensitivity` (public_safe/internal/restricted/highly_restricted) adalah atribut independen dari `verification_stage` — entitas yang sudah `active_verified` bisa tetap `highly_restricted`.

**Alasan**: Status verifikasi mengukur kelengkapan administratif; sensitivity mengukur risiko privasi. Keduanya tidak boleh dicampur agar data sensitif tidak otomatis terbuka hanya karena terverifikasi.

### DEC-007: Audit Wajib di Setiap Mutasi (Tanpa Kecuali)

**Keputusan**: Setiap operasi create/update/delete/verify menulis baris `auditEvents (collection)` dalam transaksi yang sama dengan mutasi data.

**Alasan**: Sistem ini menangani data warga yang berdampak pada bantuan sosial dan keagamaan — akuntabilitas penuh adalah kebutuhan dasar, bukan fitur tambahan.

### DEC-008: Region Code Mengikuti Struktur Kemendagri (4-Level)

**Keputusan**: `province_code`, `regency_code`, `district_code`, `village_code` mengikuti pola kode administratif resmi Indonesia, disimpan sebagai string (bukan integer) untuk mempertahankan leading zero.

**Alasan**: Interoperabilitas dengan data wilayah resmi pemerintah dan menghindari bug pembulatan/leading-zero pada integer.

### DEC-009: KV untuk Override Region/Data-Type, Bukan Sumber Kebenaran Utama

**Keputusan**: KV key `custom:regions` dan `custom:data-types` adalah override opsional di atas array statis di kode (`DEFAULT_REGION_TREE`, `DEFAULT_DATA_TYPES`), bukan sumber data utama untuk entitas registry. Data registry entity sendiri tetap di collection `registryEntities`.

**Alasan**: Wilayah dan tipe entitas relatif statis dan jarang berubah dibanding data registry — menyimpannya sebagai fallback statis + override KV menghindari kebutuhan collection/index terpisah untuk data yang jarang ditulis.

**Catatan koreksi (Juni 2026)**: Keputusan ini awalnya menyebut key KV `sikesra:public-status-cache`/`sikesra:region-tree-cache` yang tidak ada di kode, dan menyebut "D1" sebagai sumber kebenaran seolah-olah lewat SQL — keduanya salah, lihat `03.PLUGIN_ARCHITECTURE.md` § 6 dan § 8.

### DEC-010: Pertahankan Path `awcms-sikesra` (Generasi 3), Bersihkan Sisa Generasi 1/2 Secara Bertahap

**Keputusan**: Path resmi plugin/template adalah `awcms-sikesra`/`awcms-sikesraTemplate`/`awcms-sikesraTemplate-cloudflare` (Generasi 3 — lihat `03.PLUGIN_ARCHITECTURE.md` § 8a). Sisa artefak Generasi 1 (implementasi standalone yang ditinggalkan 22 Mei 2026) dan Generasi 2 (rencana penamaan `awcms-micro-sikesra` yang tidak selesai dieksekusi) dibersihkan secara bertahap, bukan dihapus sekaligus tanpa review.

**Alasan**: Generasi 3 adalah satu-satunya yang dipakai `.github/workflows/deploy-sikesra.yml` dan cocok dengan `wrangler.jsonc` produksi (worker `sikesra`, domain `sikesra.ahlikoding.com`). Mempertahankan dokumentasi yang menyebut path Generasi 1/2 sebagai aktif akan terus menyesatkan kontributor baru.

**Date:** 2026-06-16

### DEC-011: Mekanisme H1-01 — Custom Astro Route Wrapper di Template, Bukan Perubahan Plugin

**Keputusan**: Implementasi identitas terverifikasi (H1-01) dilakukan dengan membuat custom Astro route wrapper di template (`awcms-sikesraTemplate-cloudflare/src/pages/_emdash/api/plugins/awcms-sikesra/[...path].ts`) yang men-shadow route generik EmDash, bukan dengan mengubah cara plugin membaca sesi secara internal.

**Alasan teknis**: Audit framework EmDash (Juni 2026) menemukan bahwa `PluginContext`/`RouteContext` tidak meng-expose `locals.user` secara otomatis — `locals.user` hanya tersedia di layer Astro. Membuat wrapper di template adalah satu-satunya cara untuk meng-inject user ID yang terverifikasi server-side tanpa mengubah core EmDash. Alternatif lain (membaca session cookie dari dalam plugin) memerlukan akses ke auth database EmDash yang tidak seharusnya diakses plugin.

**Implikasi**: Plugin `@ahliweb/awcms-sikesra` tetap menggunakan `getRequestUserId(ctx)` yang membaca `X-Sikesra-User-Id` header — tapi header ini kini dijamin diset oleh server (wrapper), bukan klien. Plugin perlu tambah capability `users:read` untuk verifikasi lanjutan via `ctx.users.get()`.

**Date:** 2026-06-16
