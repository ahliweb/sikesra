# Master Document Index & Implementation Guide — SIKESRA

**Versi**: 1.1 — disesuaikan dengan hasil audit kode (Juni 2026)
**Tanggal**: Juni 2026
**Fungsi**: Index semua dokumen PRD + panduan cara menggunakan dokumen ini

> **Perubahan v1.1**: Audit kode langsung terhadap `src/runtime.ts` menemukan bahwa fitur registry/verifikasi/dokumen/ABAC/akses/audit **sudah terimplementasi** (39 route, 16 halaman admin), plugin ID nyata adalah `awcms-sikesra` (bukan `awcms-micro-sikesra`), dan storage memakai `PluginStorageConfig` collections (bukan tabel SQL). Seluruh dokumen di bawah ditulis ulang atau dikoreksi mengikuti temuan ini. Lihat `03.PLUGIN_ARCHITECTURE.md` § 8 untuk daftar penyimpangan lengkap dan `02.IMPLEMENTATION_BACKLOG.md` untuk backlog hardening yang menggantikan backlog "MVP dari nol" sebelumnya.

## 1. Cara Menggunakan Index Ini

### Untuk AI Agent (Junior Model)

1. Baca issue GitHub yang akan dikerjakan
2. Lihat label `epic:hardening-authz` / `epic:type-consistency` / `epic:test-coverage` untuk tahu EPIC mana
3. Baca dokumen yang **dikutip di issue** saja (biasanya 1–2 dokumen)
4. Load skill yang disebutkan di issue
5. JANGAN baca semua dokumen di index ini — terlalu banyak token
6. JANGAN asumsikan pola dari versi lama dokumen ini (tabel SQL, plugin ID `awcms-micro-sikesra`) — selalu cek `03.PLUGIN_ARCHITECTURE.md` § 8 jika ragu

### Untuk Developer Manusia

1. Mulai dari PRD utama untuk gambaran besar
2. Baca `03.PLUGIN_ARCHITECTURE.md` § 8 untuk memahami gap antara desain ideal dan kode nyata
3. Cek `02.IMPLEMENTATION_BACKLOG.md` untuk backlog hardening aktif
4. Gunakan `05.API_CONTRACT.md` dan `04.DATABASE_SCHEMA.md` sebagai referensi aktif

### Untuk Reviewer / Auditor

1. PRD → `03.PLUGIN_ARCHITECTURE.md` § 8 → `10.SECURITY_AND_PRIVACY_CHECKLIST.md` § 0
2. `16.RISK_REGISTER_AND_MITIGATION_PLAN.md` untuk risiko yang sudah terkonfirmasi
3. `06.UAT_AND_DEPLOYMENT_CHECKLIST.md` untuk kriteria sebelum deploy

## 6. Rekomendasi Proses Berikutnya (v1.1 — Hardening, Bukan Fitur Baru)

### Prioritas Tertinggi: EPIC-H1 (Otorisasi Server-Side)

1. Kerjakan H1-01 (identitas terverifikasi) lebih dulu — semua item H1 lain bergantung padanya secara logis.
2. H1-02 (permission check) dan H1-04 (validasi shape) dapat paralel dengan H1-01 selesai sebagian.
3. H1-03 (ABAC sebagai gerbang nyata) setelah H1-01/H1-02 stabil.

### Paralel: EPIC-H2 (Konsistensi Tipe) dan EPIC-H3 (Test Coverage)

Tidak bergantung pada EPIC-H1 — dapat dikerjakan kontributor berbeda secara bersamaan. Lihat `09.SPRINT_EXECUTION_PLAN.md` § Urutan yang Disarankan.

### Setelah EPIC-H1/H2/H3 Selesai

- Jalankan UAT regresi penuh (`06.UAT_AND_DEPLOYMENT_CHECKLIST.md`)
- Baru pertimbangkan fitur baru dari `13.FUTURE_ROADMAP_AND_PHASE2_BACKLOG.md`
- Pertimbangkan pilot dengan data uji (bukan data warga sungguhan) sampai hardening terverifikasi penuh oleh test regresi
