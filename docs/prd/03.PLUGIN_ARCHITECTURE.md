# Plugin Architecture — SIKESRA (Sesuai Implementasi Nyata)

**Plugin npm**: `@ahliweb/awcms-sikesra` (versi `0.1.1`)
**Plugin ID**: `awcms-sikesra` — **bukan** `awcms-micro-sikesra`
**Format**: Native EmDash Plugin (`format: "native"`)
**Direktori**: `awcmsmicro-dev/packages/plugins/awcms-sikesra/`

> **Catatan penting**: Dokumen ini ditulis ulang setelah audit kode langsung terhadap `src/runtime.ts` (2870 baris). Versi sebelumnya dari dokumen ini mengasumsikan tabel SQL mentah (`CREATE TABLE sikesra_*`) yang **tidak ada di kode**. Lihat § 8 untuk daftar lengkap penyimpangan yang ditemukan antara asumsi awal dan kode nyata.

## 1. Asal Plugin: Scaffold Contoh AWCMS-Micro

Plugin ini dibangun di atas **scaffold plugin contoh/referensi AWCMS-Micro** (lihat `package.json` keywords: `"example"`, `"tenant-ready"`; deskripsi hook lifecycle masih berbunyi `"Installed the AWCMS-Micro example plugin"` — `src/runtime.ts` § hooks). Branding dan model domain SIKESRA (8 tipe entitas, 9 stage verifikasi, region Kotawaringin Barat) sudah dilapiskan di atas scaffold tersebut, tetapi sebagian mekanisme generik dari scaffold (nama tipe `Example*`, copy "AWCMS-Micro example plugin" di audit event) belum sepenuhnya diganti.

Implikasi: saat mengerjakan plugin ini, jangan berasumsi semua nama internal (`ExampleAuditEvent`, `ExampleSettings`) sudah final/khusus-SIKESRA — beberapa di antaranya adalah sisa nama generik dari scaffold yang aman untuk direname di masa depan, tapi belum dilakukan.

## 3. Model Persistensi Nyata: `PluginStorageConfig` (Collections), Bukan SQL

Plugin **tidak pernah** menjalankan `CREATE TABLE`. Persistensi memakai abstraksi `PluginStorageConfig` milik framework EmDash/AWCMS-Micro: setiap collection dideklarasikan dengan nama dan daftar index, dan diakses lewat API `ctx.storage.<collectionName>` (`get`/`put`/`delete`/`exists`/`getMany`/`putMany`/`deleteMany`/`query`/`count`) — tidak ada SQL yang ditulis tangan oleh plugin.

```typescript
// src/runtime.ts — AWCMS_SIKESRA_STORAGE (baris ~298-353)
export const AWCMS_SIKESRA_STORAGE = {
  auditEvents: { indexes: ["timestamp", "kind", "scope", ["scope", "timestamp"]] },
  accessChangeEvents: { indexes: ["timestamp", "kind", "scope", ["scope", "timestamp"]] },
  abacChangeEvents: { indexes: ["timestamp", "kind", "scope", ["scope", "timestamp"]] },
  registryEntities: { indexes: ["code", "entityType", "sensitivity", ["entityType", "sensitivity"]] },
  abacAttributeCatalog: { indexes: ["key", "targetType", "updatedAt", ["targetType", "updatedAt"]] },
  abacPolicyRules: { indexes: ["id", "effect", "updatedAt", ["effect", "updatedAt"]] },
  supportingDocuments: { indexes: ["registryEntityId", "documentType", "sensitivity", ["registryEntityId", "sensitivity"]] },
  verificationStageState: { indexes: ["registryEntityId", "stage", "updatedAt", ["registryEntityId", "updatedAt"]] },
  abacResourceAssignments: { indexes: ["resourceId", "updatedAt"] },
  abacSubjectAssignments: { indexes: ["subjectId", "updatedAt"] },
  contentSnapshots: { indexes: ["collection", "contentId", "timestamp", ["collection", "timestamp"], ["contentId", "timestamp"]] },
  settingsState: { indexes: ["key", "updatedAt"] },
  pluginState: { indexes: ["key", "updatedAt"] },
  permissionCatalog: { indexes: ["slug", "scope", "updatedAt", ["scope", "updatedAt"]] },
  roleCatalog: { indexes: ["slug", "updatedAt"] },
  rolePermissionAssignments: { indexes: ["roleSlug", "updatedAt"] },
  userRoleAssignments: { indexes: ["userId", "updatedAt"] },
  verificationEvents: { indexes: ["registryEntityId", "stage", "createdAt", ["registryEntityId", "createdAt"]] },

```

**Tidak ada collection untuk region/wilayah atau data-type** — keduanya disimpan sebagai array statis di kode (`DEFAULT_REGION_TREE`, `DEFAULT_DATA_TYPES`) dengan override opsional di Cloudflare KV (`custom:regions`, `custom:data-types`), bukan di collection storage. Lihat § 6.

Lihat `04.DATABASE_SCHEMA.md` untuk field-level shape tiap collection.

## 4. Manifest dan Halaman Admin

`AWCMS_SIKESRA_MANIFEST` (`runtime.ts` § manifest) mendaftarkan 16 halaman admin (lihat `05.API_CONTRACT.md` § halaman) plus 3 widget (`GovernanceWidget`, `AccessRightsHealthWidget`, `AbacPolicyStatusWidget`) dan 1 field widget (`StatusBadgeField`). String permission yang dirujuk di nav manifest didefinisikan **inline** di manifest itu sendiri, bukan diimpor dari `src/permissions.ts`.

## 5. Route Architecture (39 Route Nyata)

Semua route didefinisikan di `sharedRouteEntries` (objek tunggal di `runtime.ts`), lalu diadaptasi untuk dua mode: `createNativeRoutes()` (dipakai produksi, dipanggil dari `createPlugin()` di `index.ts`) dan `createSandboxRoutes()` (mode sandbox, untuk kebutuhan marketplace masa depan — keduanya membungkus entry yang sama).

Hanya **`public/status`** yang `public: true`. Seluruh 38 route lainnya tidak punya flag `public` — lihat `05.API_CONTRACT.md` untuk daftar lengkap path + deskripsi.

## 6. Region/Wilayah dan Tipe Data: Statis di Kode, Bukan Storage Collection

```typescript
export interface AdministrativeRegion { code: string; name: string; }
export interface AdministrativeDistrict extends AdministrativeRegion { villages: AdministrativeRegion[]; }
export interface AdministrativeRegency extends AdministrativeRegion { districts: AdministrativeDistrict[]; }
export interface AdministrativeProvince extends AdministrativeRegion { regencies: AdministrativeRegency[]; }
```

`DEFAULT_REGION_TREE` adalah array statis (1 provinsi "Kalimantan Tengah"/`62` → 1 kabupaten "Kotawaringin Barat"/`6201` → 6 kecamatan dengan daftar desa). Route `regions/get` mengembalikan `(await ctx.kv.get("custom:regions")) ?? DEFAULT_REGION_TREE`; `regions/save` menulis JSON apa pun yang dikirim ke KV **tanpa validasi shape**.

Pola yang sama untuk tipe entitas: `SikesraParentType`/`SikesraSubType` (parent + sub-type, bukan flat list) dengan `DEFAULT_DATA_TYPES` berisi tepat 8 parent type yang cocok dengan domain SIKESRA (`rumah_ibadah`, `lembaga_keagamaan`, `pendidikan_keagamaan`, `lks`, `guru_agama`, `anak_yatim`, `disabilitas`, `lansia_terlantar`), masing-masing dengan sub-type. Route `data-types/get`/`data-types/save` mengikuti pola KV-override yang sama, juga tanpa validasi shape.

## 7. Hook Surface (Nyata)

```typescript
// src/runtime.ts — sharedHooks
"plugin:install"   // seed access + ABAC catalog, set state, audit "plugin.install"
"plugin:activate"  // seed ulang catalog, schedule cron "governance-summary" (tiap jam), audit
"plugin:deactivate"// cancel cron, audit
"plugin:uninstall" // cancel cron, audit (menghormati flag event.deleteData)
"content:beforeSave" / "content:afterSave"     // snapshot + audit untuk SEMUA collection EmDash (bukan hanya SIKESRA)
"content:beforeDelete" / "content:afterDelete" // audit
"content:afterPublish" / "content:afterUnpublish" // audit
"media:beforeUpload" / "media:afterUpload"     // audit
cron                // hanya menangani event "governance-summary"; menulis state + audit "cron.summary"
"page:metadata"     // inject meta tag "awcms-micro:governance-mode" + canonical link
```

Semua hook menulis audit event lewat `appendAuditEvent()` + `createAuditRecord()` yang didefinisikan di `runtime.ts` (bukan `src/audit.ts`, yang mati kode).

## 8. Penyimpangan yang Ditemukan (Wajib Dibaca Sebelum Coding)

Daftar ini hasil audit langsung kode (Juni 2026). Jangan asumsikan salah satu sisi benar tanpa konfirmasi — perbaiki secara sadar lewat PR + changelog jika ingin direkonsiliasi.

1. **Dua set level verifikasi yang tidak sinkron**: `VerificationLevel` (bucket stage) memakai `"kabupaten_admin"` dan `"tampil"`; `VerificationUserLevel` (alias `SikesraUserLevel` dari `fixtures.ts`) memakai `"kabupaten"` dan `"admin_sikesra"`. Dijembatani manual di `getAllowedVerifierLevels()` — level `"tampil"` (= stage `active_verified`) tidak mengizinkan verifier mana pun (array kosong).
2. **Tipe ABAC/permission/role berbeda** antara `fixtures.ts` (`SikesraReferenceAbacPolicy`, `SikesraReferenceAccessPermission`, `SikesraReferenceAccessRole` — ringan) dan tipe lokal nyata yang dipakai `runtime.ts` (`AbacPolicyRule`, `AccessPermission`, `AccessRole` — punya field tambahan `labelKey`, `descriptionKey`, `scope`, `updatedAt`). Hanya 3 tipe (registry entity, supporting document, verification event) yang identik antara kedua file.
3. **String permission di `src/permissions.ts` tidak dipakai produksi.** Contoh: `permissions.ts` punya `accessPreviewRead: "awcms:sikesra:access-preview:read"`, sedangkan manifest nyata di `runtime.ts` memakai `"awcms:sikesra:preview:read"` (tanpa `access-`). `permissions.ts` hanya dirujuk oleh `tests/plugin.test.ts`.
4. **`src/audit.ts` adalah dead code** — tidak diimpor oleh `index.ts`, `runtime.ts`, `admin.tsx`, atau test apa pun.
5. **Tidak ada validasi RBAC/ABAC server-side pada route mutasi apa pun.** Lihat `10.SECURITY_AND_PRIVACY_CHECKLIST.md` § temuan kritis — ini bukan detail kecil, ini gap keamanan nyata.
6. **`evaluateAbacDecision()`/`previewAccess()` tidak pernah memanggil/menghentikan mutasi nyata** — hanya dipanggil dari route `abac/preview`, `abac/enforce-demo`, `access/preview` yang bersifat simulasi read-only.
7. **Region tree dan data type tidak punya storage collection** — statis di kode + override KV tanpa validasi shape, bukan tabel/collection terkelola.
