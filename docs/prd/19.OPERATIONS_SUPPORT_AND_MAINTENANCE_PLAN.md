# Operations Support and Maintenance Plan — SIKESRA

## 1. Model Operasional

### 1.1 Tim dan Tanggung Jawab

| Peran | Tanggung Jawab Operasional |
| --- | --- |
| Admin SIKESRA kabupaten | Operasional harian, eskalasi pertama, monitoring KPI |
| Tim pengembang/pengelola repo | Perbaikan bug, rilis update plugin/template |
| Penyedia infrastruktur (Cloudflare) | Ketersediaan D1/R2/KV/Workers di tingkat platform |

### 1.2 Prinsip Operasional

- Tidak ada downtime terjadwal yang tidak diumumkan ke admin kabupaten terlebih dahulu.
- Setiap insiden dicatat sebagai audit event internal dan/atau isu GitHub untuk pelacakan.

## 2. Level Prioritas Insiden

| Level | Definisi | Contoh |
| --- | --- | --- |
| P1 — Kritis | Sistem tidak dapat diakses, atau kebocoran data | Health endpoint down, ABAC bypass terkonfirmasi |
| P2 — Tinggi | Fitur inti tidak berfungsi untuk sebagian pengguna | Verifikasi gagal di satu wilayah tertentu |
| P3 — Sedang | Bug fungsional non-blocking | Filter list registry tidak akurat |
| P4 — Rendah | Kosmetik/copy/i18n | Label belum diterjemahkan |

## 3. Prosedur Respons Insiden

### 3.1 Alur Respons

1. Insiden terdeteksi (laporan pengguna atau monitoring health endpoint).
2. Admin SIKESRA kabupaten menilai level prioritas.
3. P1/P2 dieskalasikan segera ke tim pengembang via isu GitHub berlabel prioritas tinggi.
4. P3/P4 dicatat sebagai isu biasa untuk dikerjakan pada sprint berikutnya.

### 3.2 Post-Mortem (Insiden P1)

Setiap insiden P1 didokumentasikan singkat: kronologi, akar masalah, perbaikan yang diterapkan, dan pencegahan ke depan — dicatat sebagai entri baru di `12.CHANGE_CONTROL_AND_DECISION_LOG.md` jika menghasilkan keputusan desain baru.

## 4. Backup dan Recovery

### 4.1 Backup D1

Mengikuti mekanisme backup D1 yang sudah ada di tingkat repository (`scripts/backup-sikesra-d1.sh`, dipanggil via `pnpm d1:backup:sikesra`). Backup mencakup seluruh database termasuk semua collection SIKESRA yang tersimpan sebagai JSON di D1 via `PluginStorageConfig`. Script ini membaca `sqlite_master` secara generik dan valid untuk skema apa pun yang ada di D1 produksi.

### 4.2 Backup R2

> **Catatan status (Juni 2026)**: R2 bucket `sikesra` ada di konfigurasi Cloudflare tapi **tidak ada file yang disimpan di sana** saat ini — `documents/save` (route #5) hanya menyimpan metadata di collection `supportingDocuments`, tidak ada upload file fisik (lihat RT-04 di `16.RISK_REGISTER_AND_MITIGATION_PLAN.md`). Backup R2 tidak diperlukan sampai fitur upload file diimplementasikan.

Jika fitur upload file ditambahkan di masa depan: aktifkan object versioning di tingkat konfigurasi Cloudflare R2 untuk melindungi dari penghapusan tidak sengaja.

### 4.3 Prosedur Recovery

1. Identifikasi backup D1 terbaru yang valid.
2. Restore ke environment staging terlebih dahulu untuk verifikasi (cek jumlah baris `registryEntities (collection)`, integritas relasi ke `verificationEvents (collection)`).
3. Jika valid, restore ke produksi dengan koordinasi admin SIKESRA kabupaten.
4. Smoke test pasca-restore mengikuti `06.UAT_AND_DEPLOYMENT_CHECKLIST.md` § 6.

### 4.4 Disaster Recovery

Skenario kehilangan total environment produksi: deploy ulang plugin/template ke environment Cloudflare baru, restore D1 dari backup terakhir, restore R2 dari versioning/backup terakhir, jalankan smoke test penuh sebelum mengarahkan kembali traffic.

## 5. Pemeliharaan Rutin

- Review berkala risk register (`16.RISK_REGISTER_AND_MITIGATION_PLAN.md` § 6.1).
- Review ukuran tabel `registryEntities (collection)`/`auditEvents (collection)` untuk mengantisipasi RT-04 (limit D1).
- Pembersihan audit event yang melewati retensi (cron, lihat `17.DATA_GOVERNANCE_AND_RETENTION_POLICY.md` § 4.2).

## 6. Monitoring Operasional

### 6.1 Sumber Data Monitoring

- **`public/status`** (route #1): endpoint publik untuk cek ketersediaan sistem — merespons JSON agregat jika plugin aktif
- **`access/health`** (route #28): ringkasan kesehatan katalog akses (gap permission, assignment hilang)
- **`abac/health`** (route #39): ringkasan kesehatan katalog ABAC
- **`audit/list`** (route #18): untuk deteksi pola anomali seperti lonjakan `rejected` atau aktivitas tidak wajar
- **Cloudflare Workers Analytics** dan **Wrangler tail log**: untuk error rate dan latency (tidak memerlukan endpoint khusus)

### 6.2 Kondisi Alert (Manual di MVP)

- `public/status` tidak merespons HTTP 200 lebih dari 5 menit — eskalasi P1 segera.
- Lonjakan tidak wajar pada `needs_review`/`rejected` di satu wilayah (`audit/list` filter by actor+kind) — indikasi masalah data atau pelatihan, eskalasi P3.

### 6.3 Smoke Check Ketersediaan

```text
GET /_emdash/api/plugins/awcms-sikesra/public/status
Expected HTTP 200: { "data": { ... jumlah entitas per tipe ... } }
```

> Tidak ada endpoint `/health` yang terpisah — `public/status` adalah proxy terbaik untuk ketersediaan plugin.

## 7. Prosedur Pembaruan Sistem

### 7.1 Update Platform AWCMS-Micro/EmDash

Mengikuti workflow sinkronisasi upstream yang sudah ada di root repo (`scripts/` sync workflow) — protected paths memastikan plugin/template SIKESRA tidak tertimpa saat sync.

### 7.2 Update Plugin SIKESRA

Mengikuti workflow PR standar: lint bersih, typecheck bersih, test lulus, changeset ditambahkan jika ada perubahan perilaku yang dipublikasikan, deploy mengikuti `06.UAT_AND_DEPLOYMENT_CHECKLIST.md` § 5.
