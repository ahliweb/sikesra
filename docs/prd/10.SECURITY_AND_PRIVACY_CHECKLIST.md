# Security and Privacy Checklist — SIKESRA

## 0. Temuan Kritis (Audit Kode, Juni 2026) — Baca Ini Dulu

Audit langsung terhadap `src/runtime.ts` menemukan bahwa **plugin SIKESRA saat ini tidak punya otorisasi server-side nyata**. Ini bukan asumsi atau risiko hipotetis — ini adalah kondisi kode saat ini, diverifikasi baris demi baris:

1. **Tidak ada route mutasi yang memeriksa permission.** Grep penuh terhadap `ctx.user`, pengecekan permission, atau guard otorisasi di badan handler: nol hasil untuk `registry/save`, `documents/save`, `import/promote`, `settings/save`, seluruh `access/*/save`, seluruh `abac/*/save`, `regions/save`, `data-types/save`.
2. **Pengecekan level verifier bisa dipalsukan klien.** `verification/advance`/`verification/reject` membaca `verifierLevel` dan `actor` langsung dari body request pemanggil (`getString(routeCtx.input, "verifierLevel")`), bukan dari identitas sesi yang terverifikasi.
3. **`getRequestUserId()` membaca header tak-ditandatangani** (`X-Sikesra-User-Id`) — siapa pun bisa mengirim header ini dengan nilai apa pun.
4. **ABAC tidak pernah menggerbang mutasi nyata.** `evaluateAbacDecision()` hanya dipanggil dari route `abac/preview` dan `abac/enforce-demo` — keduanya simulasi read-only. `abac/enforce-demo` mengevaluasi keputusan tapi **tidak pernah memblokir**, hanya mencatat log jika action memuat kata `"sensitive"`.
5. **`regions/save` dan `data-types/save` menulis JSON apa pun ke KV tanpa validasi shape.**

**Dampak**: setiap data yang masuk lewat plugin ini saat ini berstatus "dapat ditulis/dibaca oleh siapa pun yang bisa memanggil route plugin", terlepas dari role atau wilayah tugas mereka. Kerangka RBAC/ABAC (`permissionCatalog`, `roleCatalog`, `abacPolicyRules`, dst.) **sudah ada sebagai data model dan UI admin**, tetapi belum disambungkan sebagai gerbang nyata di jalur mutasi.

**Jangan deploy plugin ini untuk menangani data warga sungguhan (terutama kategori sensitif: anak yatim, disabilitas, lansia terlantar) sampai EPIC-H1 di `02.IMPLEMENTATION_BACKLOG.md` selesai.** Bagian-bagian di bawah (§ 1-9) mendokumentasikan kondisi *seharusnya* (target hardening) — bandingkan dengan § 0 ini untuk tahu gap-nya.

## 1. Pendekatan Keamanan (Target, Belum Tercapai)

### 1.1 Prinsip Target

- **Least privilege**: permission granular per route, bersumber dari `permissionCatalog`/`roleCatalog`/`userRoleAssignments` (collection yang sudah ada datanya, tapi belum dibaca route mutasi).
- **Defense in depth**: RBAC (aksi apa) + ABAC (data mana) sebagai dua lapis — kerangka datanya ada, gerbangnya belum.
- **Default deny**: `evaluateAbacDecision()` sudah berperilaku default-deny secara logika, tapi karena tidak pernah dipanggil di jalur mutasi, default-deny ini tidak berlaku saat ini.
- **Audit semua mutasi**: ini **satu-satunya prinsip yang sudah terpenuhi** — setiap route mutasi memang menulis ke `auditEvents`/`accessChangeEvents`/`abacChangeEvents`.

### 1.2 Referensi Regulasi

UU No. 27/2022 tentang Pelindungan Data Pribadi (PDP) — relevan untuk data individu (guru agama, anak yatim, disabilitas, lansia terlantar). Status kepatuhan saat ini: **belum memenuhi**, karena tidak ada kontrol akses nyata terhadap data tersebut (lihat § 0).

## 2. Autentikasi

Plugin tidak mengimplementasikan autentikasi sendiri secara eksplisit untuk route-nya — autentikasi seharusnya ditangani sesi EmDash core, tetapi route plugin **tidak terbukti membaca identitas dari sesi tersebut** (lihat § 0 poin 2-3). Sebelum hardening, anggap setiap route plugin sebagai endpoint tanpa autentikasi efektif.

## 3. Otorisasi RBAC (Target vs Nyata)

### 3.1 Data Model yang Sudah Ada

`permissionCatalog`, `roleCatalog`, `rolePermissionAssignments`, `userRoleAssignments` — collection ini ada, di-seed saat install/activate (`ensureAccessCatalogSeeded()`), dan punya UI admin penuh (`access/permissions`, `access/roles`, `access/matrix`). String permission nyata didefinisikan inline di manifest `runtime.ts` — **berbeda** dari `src/permissions.ts` (lihat `03.PLUGIN_ARCHITECTURE.md` § 8 poin 3).

### 3.2 Checklist Hardening (EPIC-H1 di `02.IMPLEMENTATION_BACKLOG.md`)

- [ ] H1-01: identitas pemanggil berasal dari sesi EmDash terverifikasi, bukan header/body request
- [ ] H1-02: setiap route `*/save` memeriksa permission yang sesuai sebelum menulis
- [ ] Setelah H1-02: tambah test yang membuktikan permission ditolak dikembalikan sebagai error, bukan diam-diam sukses

## 4. Otorisasi ABAC (Target vs Nyata)

### 4.1 Data Model yang Sudah Ada

`abacAttributeCatalog`, `abacPolicyRules`, `abacSubjectAssignments`, `abacResourceAssignments` — ada, dengan UI admin penuh dan fungsi evaluasi (`evaluateAbacDecision()`) yang sudah benar secara logika (default-deny, priority-based matching).

### 4.2 Checklist Hardening (EPIC-H1)

- [ ] H1-03: `evaluateAbacDecision()` dipanggil di `registry/save`, `verification/advance`, `verification/reject`, `documents/save` sebelum menulis, dan hasil `deny` benar-benar menghentikan mutasi
- [ ] Test: subject dari wilayah A tidak bisa menulis/membaca resource wilayah B setelah H1-03 selesai

## 5. Keamanan Data Input

### 5.1 Validasi Shape pada Route KV-Override

`regions/save` dan `data-types/save` menulis JSON tanpa validasi (§ 0 poin 5). Checklist: tambahkan skema Zod untuk `AdministrativeProvince[]` dan `SikesraParentType[]` (H1-04).

### 5.2 SQL Injection

Tidak relevan langsung — plugin tidak menulis SQL manual (storage lewat collections). Aturan root `AGENTS.md` § SQL tetap berlaku untuk kode EmDash core yang mungkin disentuh tidak langsung.

### 5.3 XSS Prevention

`publicSummary`, `label`, dan field teks bebas lain harus di-escape di sisi render (default React/Astro escaping) — tidak ditemukan penggunaan `dangerouslySetInnerHTML` di audit awal, tapi belum diverifikasi penuh untuk seluruh `admin.tsx` (4553 baris).

## 6. Keamanan Penyimpanan

Region tree dan data-type catalog memakai KV tanpa collection storage terkelola (lihat `03.PLUGIN_ARCHITECTURE.md` § 6) — ini bukan kerentanan langsung, tapi berarti tidak ada index/query terstruktur untuk audit konsistensi wilayah.

## 7. Keamanan Jaringan

Endpoint dev-only mengikuti pola `import.meta.env.DEV` standar root `AGENTS.md` — tidak diverifikasi khusus untuk plugin ini dalam audit ini.

## 8. Audit dan Monitoring Keamanan

### 8.1 Yang Sudah Berfungsi

Setiap route mutasi menulis ke `auditEvents` (lifecycle, content, media) atau `accessChangeEvents`/`abacChangeEvents` (perubahan RBAC/ABAC). `audit/list` menyediakan paginasi cursor (maks 50) untuk membaca log ini.

### 8.2 Yang Belum Ada

Tidak ada alerting otomatis. Tidak ada cara membedakan "mutasi sah" dari "mutasi oleh pemanggil tanpa otorisasi" di dalam audit log itu sendiri — karena tidak ada otorisasi yang dicek, semua mutasi tercatat sama saja kualitasnya.

## 9. Privasi Data PII

### 9.1 Field Berisiko

Entitas tipe `guru_agama`, `anak_yatim`, `disabilitas`, `lansia_terlantar` membawa data individu sensitif. `sensitivity` field ada di model data (`registryEntities`, `supportingDocuments`), tapi **tidak ada enforcement yang membatasi siapa boleh membaca level `highly_restricted`** — field ini saat ini bersifat deskriptif (metadata), bukan kontrol akses aktif.

### 9.2 Agregat Publik

`public/status` (satu-satunya route publik) menerapkan small-cell suppression berdasarkan `smallCellThreshold` di settings — ini **sudah berfungsi** sebagai kontrol privasi nyata untuk endpoint publik, berbeda dari gap RBAC/ABAC di route privat.
