# User Manual dan SOP — SIKESRA

## 1. Tentang Dokumen Ini

Panduan operasional untuk pengguna akhir SIKESRA: petugas desa/kelurahan, verifikator kecamatan, verifikator SOPD, verifikator/admin kabupaten. Dokumen ini bersifat draft hidup — diperbarui setiap UI berubah signifikan (lihat `12.CHANGE_CONTROL_AND_DECISION_LOG.md`).

> **Catatan penting**: SOP di bawah menjelaskan alur kerja yang dimaksudkan oleh desain UI. Audit kode (Juni 2026, lihat `10.SECURITY_AND_PRIVACY_CHECKLIST.md` § 0) menemukan bahwa pembatasan akses berbasis role/wilayah yang dijelaskan di sini **belum ditegakkan di server** — gunakan plugin ini untuk data uji/latihan sampai backlog hardening (`02.IMPLEMENTATION_BACKLOG.md` EPIC-H1) selesai, jangan untuk data warga sungguhan kategori sensitif.

## 2. Akses Sistem

### 2.1 URL Sistem

Admin SIKESRA diakses melalui `/_emdash/admin/` pada domain instansi yang menjalankan AWCMS-Micro + plugin SIKESRA. Halaman publik agregat (jika diaktifkan) diakses melalui rute publik template.

### 2.2 Login

Login menggunakan mekanisme autentikasi core EmDash (dikelola platform, bukan plugin SIKESRA). Setiap pengguna memiliki satu akun dengan satu atau lebih role SIKESRA (`userRoleAssignments (collection)`).

### 2.3 Dashboard Setelah Login

Setelah login, pengguna melihat halaman `overview` SIKESRA berisi ringkasan jumlah entitas per tipe, status verifikasi yang menunggu tindakan pengguna, dan tautan cepat ke modul lain sesuai permission.

### 2.4 Logout

Logout mengikuti mekanisme core EmDash (tombol logout pada navigasi utama).

### 2.5 Reset Password

Reset password ditangani oleh core EmDash, bukan oleh plugin SIKESRA.

## 3. SOP Petugas Desa/Kelurahan

### 3.1 Membuat Entitas Registry Baru

1. Buka modul **Registri**.
2. Klik tombol tambah, pilih `entity_type` (salah satu dari 8 tipe: rumah ibadah, lembaga keagamaan, pendidikan keagamaan, LKS, guru agama, anak yatim, disabilitas, lansia terlantar).
3. Isi `label`, region (province/regency/district/village code — otomatis terisi sesuai wilayah tugas petugas), dan `public_summary` (ringkasan aman tanpa identitas pribadi untuk kategori sensitif).
4. Simpan sebagai `draft`.

### 3.2 Mengunggah Dokumen Pendukung

1. Dari halaman detail entitas, buka tab **Dokumen**.
2. Unggah file (maks. 10MB), pilih `document_type` dan `sensitivity`.
3. Dokumen tersimpan dan tertaut ke entitas.

### 3.3 Mengajukan Verifikasi

1. Setelah data dan dokumen lengkap, klik **Ajukan Verifikasi** untuk memindahkan stage `draft` → `submitted_village`.
2. Setelah diajukan, data hanya dapat diedit terbatas sampai dikembalikan (`needs_review`) atau disetujui.

### 3.4 Menindaklanjuti Hasil Verifikasi

- Jika hasil `needs_review`: baca catatan verifikator, perbaiki data/dokumen, ajukan ulang.
- Jika hasil `rejected`: entitas berhenti di stage tersebut; petugas dapat membuat entitas baru jika data ternyata salah secara fundamental.

## 4. SOP Verifikator Kecamatan

### 4.1 Melihat Antrian Verifikasi

Modul **Verifikasi** menampilkan daftar entitas dengan stage `submitted_village` di wilayah kecamatan verifikator (difilter otomatis oleh ABAC).

### 4.2 Memproses Verifikasi

1. Buka detail entitas, periksa data dan dokumen pendukung.
2. Pilih hasil: `approved` (lanjut ke `verified_village` lalu otomatis tersedia untuk diajukan ke `submitted_district`), `needs_review` (kembali ke petugas desa dengan catatan), atau `rejected`.
3. Isi `notes` — wajib diisi untuk `needs_review` dan `rejected`.

## 5. SOP Verifikator SOPD

### 5.1 Antrian Verifikasi SOPD

Modul **Verifikasi** menampilkan entitas stage `submitted_sopd` yang relevan dengan domain SOPD tersebut (mis. SOPD sosial untuk LKS/anak yatim/disabilitas/lansia, SOPD keagamaan untuk rumah ibadah/lembaga keagamaan/guru agama).

### 5.2 Proses Verifikasi Substansi

Verifikator SOPD melakukan review substansi (bukan hanya administratif) sebelum meneruskan ke `verified_sopd` → `submitted_regency`.

## 6. SOP Admin/Verifikator Kabupaten

### 6.1 Verifikasi Akhir

Entitas dengan stage `submitted_regency` diverifikasi akhir oleh admin/verifikator kabupaten menjadi `active_verified` — status final yang membuat entitas tampil di agregat publik (jika tidak `suppressed` karena sensitivity).

### 6.2 Kelola Akses (Roles & Permissions)

Modul **Akses**: admin SIKESRA dapat membuat role baru, mengubah assignment permission per role, dan menetapkan role ke pengguna dengan `region_scope` tertentu.

### 6.3 Kelola Kebijakan ABAC

Modul **ABAC**: admin dapat melihat/menambah atribut dan policy rule, serta menjalankan **ABAC Preview** untuk menguji kombinasi subject/resource sebelum mengaktifkan policy baru di produksi.

### 6.4 Kelola Pengaturan Plugin

Modul **Pengaturan**: konfigurasi umum plugin (lihat `04.DATABASE_SCHEMA.md` § KV Keys untuk `sikesra:settings`).

## 7. SOP Auditor (Read-Only)

### 7.1 Mengakses Audit Trail

Modul **Audit Log** menampilkan seluruh event dengan filter `kind`, `actor`, dan rentang tanggal. Akses bersifat read-only (permission `audit:read`).

### 7.2 Menelusuri Riwayat Entitas

Dari halaman detail entitas registry, tab **Riwayat Verifikasi** menampilkan seluruh `verificationEvents (collection)` terkait secara kronologis.

## 8. Panduan Troubleshooting Umum

| Masalah | Kemungkinan Sebab | Solusi |
| --- | --- | --- |
| "Akses ditolak" saat membuka entitas | ABAC region scope tidak cocok | Periksa `region_scope` pada `userRoleAssignments (collection)` milik pengguna |
| Tidak bisa mengajukan verifikasi | Dokumen pendukung belum lengkap | Lengkapi dokumen wajib sebelum submit |
| Transisi stage ditolak (`SIKESRA_LEVEL_MISMATCH`) | Level pengguna tidak sesuai stage target | Pastikan verifikator sesuai `STAGE_VERIFIER_LEVEL` (lihat `skills/sikesra-data-d1/SKILL.md`) |
| Tidak bisa unduh dokumen | Sensitivity terlalu tinggi untuk level pengguna | Hubungi admin SIKESRA untuk eskalasi akses |

## 9. Kontak Dukungan

Dukungan teknis ditangani oleh tim pengelola SIKESRA kabupaten (kontak internal — lihat saluran komunikasi resmi kabupaten, tidak dipublikasikan di dokumen ini).
