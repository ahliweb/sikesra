# Data Governance and Retention Policy — SIKESRA

## 1. Kerangka Tata Kelola Data

### 1.1 Prinsip Dasar

- Setiap data yang masuk SIKESRA punya satu pemilik wilayah (region) dan satu klasifikasi sensitivity yang jelas.
- Klasifikasi sensitivity ditentukan oleh `entity_type` secara default, dapat dinaikkan manual tapi tidak diturunkan tanpa alasan tercatat.
- Data individu sensitif (anak yatim, disabilitas, lansia terlantar, guru agama) tidak pernah ditampilkan di luar baris yang dilindungi ABAC.

### 1.2 Peran dalam Tata Kelola Data

| Peran | Tanggung Jawab Data |
| --- | --- |
| Petugas desa/kelurahan | Akurasi input awal di wilayahnya |
| Verifikator kecamatan/SOPD/kabupaten | Validasi kebenaran data sebelum naik stage |
| Admin SIKESRA | Klasifikasi sensitivity, kebijakan ABAC, retensi |

## 2. Klasifikasi Data

### 2.1 Klasifikasi Default per Tipe Entitas

| `entity_type` | Sensitivity Default |
| --- | --- |
| `rumah_ibadah` | `public_safe` |
| `lembaga_keagamaan` | `internal` |
| `pendidikan_keagamaan` | `internal` |
| `lks` | `restricted` |
| `guru_agama` | `restricted` |
| `anak_yatim` | `highly_restricted` |
| `disabilitas` | `restricted` atau `highly_restricted` (kasus berisiko tinggi) |
| `lansia_terlantar` | `restricted` atau `highly_restricted` (kasus berisiko tinggi) |

### 2.2 Aturan Perubahan Klasifikasi

Menaikkan sensitivity selalu diperbolehkan oleh verifikator level mana pun yang memiliki akses entitas. Menurunkan sensitivity hanya dapat dilakukan oleh `admin_sikesra` dan harus tercatat sebagai audit event dengan alasan di `metadata`.

## 3. Inventaris Data

| Tabel | Jenis Data | Klasifikasi |
| --- | --- | --- |
| `registryEntities (collection)` | Data entitas registry (campuran publik dan sensitif) | Bervariasi per baris (`sensitivity` field) |
| `supportingDocuments (collection)` | Dokumen pendukung, dapat berisi identitas individu | Bervariasi per baris |
| `verificationEvents (collection)` | Riwayat keputusan verifikasi | `internal` |
| `auditEvents (collection)` | Jejak audit seluruh sistem | `internal` |

### 3.1 Catatan Penting

`public_summary` pada entitas adalah satu-satunya field yang berpotensi muncul di permukaan publik — field ini harus selalu ditulis seolah-olah akan dibaca publik, terlepas dari `sensitivity` baris induknya.

## 4. Kebijakan Retensi

### 4.1 Retensi per Jenis Data (MVP)

| Jenis Data | Retensi Minimum | Catatan |
| --- | --- | --- |
| `registryEntities (collection)` | Selama entitas masih relevan secara administratif | Tidak ada penghapusan otomatis di MVP |
| `supportingDocuments (collection)` | Sama dengan entitas induk | Dihapus manual jika entitas dihapus oleh admin |
| `verificationEvents (collection)` | Permanen (riwayat tidak dihapus) | Dasar akuntabilitas verifikasi berjenjang |
| `auditEvents (collection)` | Minimal 1 tahun (indikatif, dapat diperpanjang sesuai regulasi daerah) | Pembersihan via cron setelah melewati retensi |

### 4.2 Catatan MVP vs Fase Berikutnya

MVP belum mengimplementasikan penghapusan otomatis audit event. Satu-satunya cron yang aktif di plugin adalah `"governance-summary"` (dijadwalkan setiap jam — `"0 * * * *"`) via `ctx.cron.schedule()` di hook `plugin:activate`. Cron ini menghasilkan ringkasan governance berkala; tidak menangani retensi audit. Retensi audit penuh (penghapusan baris `auditEvents` yang melebihi batas) adalah kandidat Phase 2.

## 5. Siklus Hidup Entitas Registry

Entitas mengikuti siklus `draft` → ... → `active_verified`. Tidak ada siklus "arsip" formal di MVP — entitas yang sudah tidak relevan (misal rumah ibadah yang sudah tidak beroperasi) ditandai melalui mekanisme penghapusan/penonaktifan oleh admin, bukan transisi stage.

## 6. Prosedur Retensi

### 6.1 Monitoring Retensi

Bagian dari tugas operasional rutin admin SIKESRA (lihat `19.OPERATIONS_SUPPORT_AND_MAINTENANCE_PLAN.md`) — bukan proses otomatis penuh di MVP.

### 6.2 Penanganan Data yang Melewati Retensi

Untuk audit event yang melewati retensi, cron pembersihan menghapus baris terkait. Untuk data registry yang sudah tidak relevan, admin melakukan soft handling (nonaktifkan, bukan hard delete) agar riwayat verifikasi tidak hilang.

## 7. Penanganan PII

### 7.1 Inventaris Field PII

Nama individu dan detail kondisi pribadi pada entitas `guru_agama`, `anak_yatim`, `disabilitas`, `lansia_terlantar`, serta identitas pada `supporting_documents` terkait.

### 7.2 Prinsip Akses PII

Akses ke field PII selalu melalui kombinasi RBAC (permission) + ABAC (region scope) + sensitivity check (level pengguna ≥ level sensitivity data). Tidak ada jalur akses PII yang melewati ketiga lapis ini.

> **Status implementasi (Juni 2026)**: Ketiga lapis ini adalah TARGET desain, belum berfungsi sebagai gerbang server nyata. Kondisi saat ini: tidak ada lapis yang aktif sebagai enforcement. Backlog EPIC-H1 (`02.IMPLEMENTATION_BACKLOG.md`) adalah pekerjaan untuk mengaktifkan lapis ini satu per satu. Jangan mendokumentasikan lapis ini kepada pengguna akhir sebagai fitur yang sudah aktif.

### 7.3 Hak Subjek Data

Mengacu UU PDP — individu yang datanya tercatat berhak meminta koreksi data melalui petugas desa/kelurahan setempat (jalur administratif, bukan endpoint self-service di MVP).

### 7.4 Prosedur Insiden Data

Jika terjadi indikasi kebocoran data (lihat risiko RP-01/RP-02 di `16.RISK_REGISTER_AND_MITIGATION_PLAN.md`), admin SIKESRA kabupaten menjadi titik eskalasi pertama untuk investigasi internal.

## 8. Kepatuhan Regulasi

- UU No. 27/2022 tentang Pelindungan Data Pribadi.
- Regulasi daerah Kabupaten Kotawaringin Barat terkait pengelolaan data kesejahteraan sosial (jika ada, dirujuk di luar dokumen ini sesuai kebijakan instansi).

## 9. Audit Kualitas Data

### 9.1 Validasi Konsistensi Region Code

Region code (`provinceCode`/`regencyCode`/`districtCode`/`villageCode`) divalidasi terhadap struktur hierarki yang valid saat input — mencegah kombinasi kode yang tidak konsisten (lihat `21.ADDENDUM_MASTER_DATA_WILAYAH.md`).

> **Status implementasi (Juni 2026)**: Validasi ini adalah TARGET. Kondisi saat ini: route `registry/save` tidak memvalidasi region code terhadap region tree — kode apa pun yang dikirim klien akan tersimpan. Validasi ini adalah bagian dari backlog H1-04 (lihat `02.IMPLEMENTATION_BACKLOG.md`).

### 9.2 Completeness Check

Modul Reports dapat menyoroti entitas yang kekurangan dokumen pendukung wajib sebagai indikator kualitas data yang belum lengkap.
