# Future Roadmap and Phase 2 Backlog — SIKESRA

## 1. Visi Jangka Panjang

SIKESRA bertujuan menjadi sumber data tunggal kesejahteraan sosial dan keagamaan Kabupaten Kotawaringin Barat yang terverifikasi berjenjang, dengan agregat publik yang aman dipublikasikan, dan basis untuk perencanaan kebijakan sosial-keagamaan berbasis data.

### 1.1 Timeline Visi (Indikatif)

```text
MVP (Sprint 0-4)        : Registry + verifikasi berjenjang + ABAC + audit dasar
Phase 2 (pasca-MVP)     : Integrasi data wilayah resmi, laporan lanjutan, notifikasi
Phase 3 (jangka panjang): Integrasi sistem pemerintah lain, dashboard analitik lanjutan
```

## 2. Phase 1 — MVP (Sprint 0-4) [Cakupan Saat Ini]

### 2.1 Ruang Lingkup MVP

- Registry 8 tipe entitas dengan verifikasi 9-stage.
- RBAC + ABAC region-aware.
- Dokumen pendukung dengan kontrol sensitivity.
- Audit trail lengkap.
- Agregat publik aman (suppression otomatis untuk data sensitif).
- Admin UI lengkap (Kumo + Lingui, RTL-safe).

### 2.2 Deliverable MVP

Lihat `09.SPRINT_EXECUTION_PLAN.md` untuk rincian per sprint dan exit criteria.

## 3. Phase 2 — Pasca-MVP (Kandidat, Belum Dikomit Tanggal)

### 3.1 Integrasi Data Wilayah Resmi (Kemendagri)

Sinkronisasi `province_code`/`regency_code`/`district_code`/`village_code` dengan sumber data wilayah resmi terbaru, menggantikan input manual saat seed. Lihat `21.ADDENDUM_MASTER_DATA_WILAYAH.md` untuk desain awal.

### 3.2 Notifikasi In-App dan Email

Notifikasi ke petugas/verifikator saat ada entitas baru yang menunggu tindakan mereka (saat ini pengguna harus membuka modul Verifikasi secara manual).

### 3.3 Laporan dan Ekspor Lanjutan

Ekspor CSV/PDF per wilayah/tipe entitas untuk kebutuhan rapat koordinasi pemerintah daerah, di luar agregat publik yang sudah ada di MVP.

### 3.4 Riwayat Perubahan Field-Level

Saat ini audit mencatat event (apa yang terjadi), bukan diff field-level penuh (apa yang berubah dari nilai A ke nilai B). Phase 2 dapat menambah snapshot before/after pada `metadata` audit event.

### 3.5 Multi-Kabupaten / Multi-Tenant

MVP diasumsikan satu instalasi = satu kabupaten. Dukungan multi-tenant (beberapa kabupaten dalam satu instalasi) adalah perubahan arsitektur besar dan harus melalui Discussion + change control formal sebelum dikerjakan.

## 4. Phase 3 — Konseptual (Belum Direncanakan Detail)

- Dashboard analitik lanjutan (tren verifikasi per waktu, peta panas wilayah).
- Integrasi dengan sistem informasi kesejahteraan sosial nasional (jika tersedia API resmi).
- Fitur AI-assisted untuk membantu verifikator memprioritaskan antrian berdasarkan risiko/usia data.

## 6. Kriteria Masuk Phase 2

Sebuah item baru boleh dikerjakan sebagai Phase 2 jika:

1. MVP sudah live dan stabil (kriteria go-live di `06.UAT_AND_DEPLOYMENT_CHECKLIST.md` terpenuhi).
2. Item tersebut tidak mengubah invariant inti MVP tanpa melalui change control (`12.CHANGE_CONTROL_AND_DECISION_LOG.md`).
3. Ada kebutuhan nyata dari pengguna (bukan asumsi) yang mendasarinya.

## 7. Prinsip Roadmap

- **MVP dulu, sempurna kemudian.** Jangan menunda Sprint 0-4 untuk menunggu desain Phase 2 selesai.
- **Aditif, bukan destruktif.** Fitur Phase 2 menambah, bukan mengganti, kontrak API/skema yang sudah dirilis — kecuali ada keputusan breaking change yang eksplisit.
- **Privasi tetap baris pertama.** Setiap fitur Phase 2 yang menyentuh data sensitif harus melewati ulasan `10.SECURITY_AND_PRIVACY_CHECKLIST.md` sebelum dikerjakan.
