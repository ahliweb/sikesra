# Addendum: Profil Personil Verifikator — SIKESRA (Usulan, Belum Diimplementasikan)

> Versi sebelumnya dokumen ini menulis skema SQL `sikesra_personnel_profiles`. Tidak ada implementasi seperti ini di kode — dokumen ini sekarang murni usulan desain (opsional), ditulis ulang memakai pola storage collection yang nyata dipakai plugin ini, bukan SQL.

## 1. Status Saat Ini

Identitas verifikator yang tercatat di `verificationEvents` adalah field `actor` (string bebas) dan opsional `verifierLevel`/`verifierRegionScope`/`verifierOrgScope` — **tanpa profil personil terstruktur** (nama, jabatan, instansi). `userRoleAssignments` (collection yang ada) hanya menyimpan `userId` ↔ `roleSlug`, tidak ada field nama/jabatan.

**Penting**: seperti dicatat di `10.SECURITY_AND_PRIVACY_CHECKLIST.md` § 0, `actor` saat ini berasal dari klaim sisi klien, bukan identitas terverifikasi. Addendum ini mengusulkan struktur data tambahan untuk profil, **bukan solusi untuk gap otorisasi** — itu domain EPIC-H1 di `02.IMPLEMENTATION_BACKLOG.md`.

## 2. Usulan Collection (Jika Dibutuhkan ke Depan)

Jika kabupaten membutuhkan tampilan "Diverifikasi oleh: [Nama, Jabatan]" yang lebih formal daripada `actor` mentah, usulan collection baru mengikuti pola `PluginStorageConfig` yang sudah dipakai plugin ini (lihat `03.PLUGIN_ARCHITECTURE.md` § 3) — **bukan tabel SQL**:

```typescript
// Usulan — BELUM ada di AWCMS_SIKESRA_STORAGE
personnelProfiles: {
  indexes: ["userId", "orgUnit", "updatedAt"],
}

interface PersonnelProfile {
  userId: string;
  fullName: string;
  positionTitle?: string;
  level: "desa_kelurahan" | "kecamatan" | "sopd" | "kabupaten" | "admin_sikesra";
  updatedAt: string;
}
```

Ini adalah usulan untuk didiskusikan (`12.CHANGE_CONTROL_AND_DECISION_LOG.md` § 2), bukan rencana kerja yang sudah disetujui.

## 3. Snapshot Identitas di Verification Event (Pola yang Sudah Mendukung, Belum Dipakai)

`verificationEvents` sudah punya field opsional yang bisa menampung snapshot ringan (`verifierLevel`, `verifierRegionScope`, `verifierOrgScope`) tanpa perlu collection baru — jika kebutuhannya hanya menampilkan level/scope verifikator (bukan nama lengkap), field yang sudah ada ini cukup. Collection `personnelProfiles` baru hanya dibutuhkan jika nama/jabatan harus ditampilkan.

## 4. Mengapa Ini Bukan Prioritas

Sebelum menambah collection baru untuk profil personil, EPIC-H1 (otorisasi server-side nyata) harus selesai dulu — menambah data profil di atas fondasi tanpa identitas terverifikasi hanya menambah permukaan data yang sama tidak terlindunginya. Lihat `02.IMPLEMENTATION_BACKLOG.md` untuk urutan prioritas.
