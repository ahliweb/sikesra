## 1. Tujuan dan Cakupan

Dokumen ini mendefinisikan skenario UAT dan checklist deployment untuk plugin `@ahliweb/awcms-sikesra` (plugin ID nyata: `awcms-sikesra`), dengan fokus memverifikasi backlog hardening EPIC-H1/H2/H3, bukan fitur baru ŌĆö fitur registry/verifikasi/dokumen/ABAC/akses/audit sudah ada di kode.

## 2. Struktur UAT

### 2.1 Pendekatan UAT

UAT dijalankan per item EPIC-H1/H2/H3 setelah masing-masing selesai, ditambah satu siklus regresi penuh atas fitur yang sudah ada (registry, verifikasi, dokumen, ABAC, akses, audit) untuk memastikan hardening tidak merusak fungsi yang sudah berjalan.

### 2.2 Tim UAT

| Peran | Tanggung Jawab |
| --- | --- |
| Admin SIKESRA (kabupaten) | Validasi pengaturan, akses, ABAC |
| Verifikator SOPD/kecamatan | Validasi alur verifikasi per level |
| Petugas Desa/Kelurahan | Validasi input dan submit data |
| Penguji independen (AI/manual) | Eksekusi skenario di bawah dan lapor temuan |

### 2.3 Lingkungan UAT

D1 lokal (`wrangler d1` local) atau in-memory binding untuk skenario non-Cloudflare-spesifik. Tidak ada `dev-bypass` khusus plugin SIKESRA yang terverifikasi ŌĆö gunakan mekanisme dev-bypass standar EmDash core (`/_emdash/api/setup/dev-bypass`, `/_emdash/api/auth/dev-bypass`) sesuai root `AGENTS.md`.

### 2.4 Data Test

Gunakan fixture referensi dari `src/fixtures.ts` (`SIKESRA_REFERENCE_FIXTURES`): 3 registry entity, 4 supporting document, 3 verification event.

### 2.5 Klasifikasi Bug

| Severity | Definisi |
| --- | --- |
| Blocker | Mutasi tanpa otorisasi berhasil tembus setelah hardening seharusnya menolak; data hilang/salah |
| Major | Fitur berjalan tapi hasil salah pada edge case (mis. level `tampil` di stage `active_verified`) |
| Minor | Kosmetik, copy, i18n yang belum diterjemahkan |

## 3. Skenario UAT per Item Hardening (EPIC-H1/H2/H3)

### 3.1 H1-01/H1-02 ŌĆö Identitas dan Permission Check

- Request ke route `*/save` tanpa identitas sah ditolak (bukan diam-diam sukses seperti sebelum hardening).
- Request dengan identitas sah tapi tanpa permission yang sesuai ditolak.
- Request dengan identitas + permission yang sesuai tetap berhasil seperti sebelumnya (regresi negatif).

### 3.2 H1-03 ŌĆö ABAC sebagai Gerbang Nyata

- Subject dari `village_code` A tidak bisa menulis/membaca resource `village_code` B setelah `evaluateAbacDecision()` disambungkan ke `registry/save`/`verification/advance`.
- Policy `deny` benar-benar menghentikan mutasi (bukan hanya tercatat di log seperti `abac/enforce-demo` sebelumnya).

### 3.3 H1-04 ŌĆö Validasi Shape Region/Data-Type

- `regions/save` dengan payload yang tidak sesuai `AdministrativeProvince[]` ditolak dengan error validasi, bukan tersimpan begitu saja.
- `data-types/save` dengan payload yang tidak sesuai `SikesraParentType[]` ditolak.

### 3.4 H2-01 ŌĆö Konsistensi Level Verifikasi

- Setelah rekonsiliasi `VerificationLevel`/`VerificationUserLevel`, perilaku stage `active_verified` (level `tampil`) terdefinisi jelas dan terdokumentasi ŌĆö bukan diam-diam tidak punya verifier.

### 3.5 Regresi Fitur yang Sudah Ada (Wajib Sebelum dan Sesudah Hardening)

- `registry/save`/`registry/list` tetap berfungsi untuk 8 tipe entitas.
- `verification/advance`/`verification/reject` tetap mengikuti 9 stage berurutan.
- `public/status` tetap menerapkan small-cell suppression berdasarkan `smallCellThreshold`.
- `audit/list` tetap mencatat setiap mutasi.

## 5. Deployment Checklist

### 5.1 Pre-Deployment

- [ ] Changeset dibuat jika perilaku publik plugin berubah.
- [ ] `pnpm build` di `awcmsmicro-dev/` sukses tanpa error.
- [ ] Binding D1/KV yang dipakai plugin sudah terkonfigurasi di environment target (lihat `wrangler.jsonc` template Cloudflare untuk ID binding aktual ŌĆö jangan asumsikan nama/ID tanpa mengecek).

### 5.2 Deployment Steps

```text
# Pastikan di branch yang sudah lulus UAT
git status

# Build seluruh workspace
pnpm build

# Tidak ada migrasi SQL terpisah ŌĆö collection storage dibuat otomatis
# oleh hook plugin:install/plugin:activate saat plugin pertama kali aktif.

# Verifikasi plugin aktif dengan memanggil route non-publik yang sudah
# diketahui (mis. dashboard/summary) lewat admin UI yang sudah login,
# karena TIDAK ADA endpoint health publik di plugin ini.
```

### 5.3 Catatan Penting

Plugin ini **tidak punya endpoint health publik**. Verifikasi deploy berhasil dilakukan lewat UI admin (login ŌåÆ buka halaman `overview`), bukan lewat `curl` ke endpoint publik ŌĆö satu-satunya route publik adalah `public/status`, yang mengembalikan agregat, bukan status kesehatan plugin.

## 6. Smoke Test Pasca-Deploy

- [ ] Login sebagai admin SIKESRA, akses halaman `overview`.
- [ ] Buat satu entitas registry uji, verifikasi muncul di `registry/list`.
- [ ] Jalankan satu transisi verifikasi (`draft` ŌåÆ `submitted_village`), cek `audit/list` mencatatnya.
- [ ] Cek `public/status` mengembalikan agregat tanpa membocorkan data sensitif.
- [ ] Rollback plan: revert ke deployment sebelumnya via Cloudflare dashboard jika smoke test gagal.
