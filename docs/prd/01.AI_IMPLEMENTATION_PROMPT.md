**Status nyata**: Plugin **sudah terimplementasi secara substansial** (39 route, 16 halaman admin) tapi dibangun di atas scaffold plugin contoh/referensi AWCMS-Micro dan **belum punya otorisasi server-side nyata**. Lihat § 2 HR-07 — ini adalah aturan keras yang paling penting di dokumen ini.

## 2. Aturan Keras (Hard Rules)

### HR-01: Plugin-Owned Storage — Jangan Sentuh Core Tables

Akses data lewat `ctx.storage.<collectionName>` (collections, lihat `04.DATABASE_SCHEMA.md`), bukan SQL mentah dan bukan tabel `_emdash_*`.

### HR-02: Jangan Modifikasi Core AWCMS-Micro

Perluas lewat `createSharedHooks()`/route baru di `src/runtime.ts`, jangan ubah `awcmsmicro-dev/packages/core/`.

### HR-03: Semua Mutasi WAJIB Tulis Audit Event — Pakai Helper di `runtime.ts`, Bukan `src/audit.ts`

```typescript
// ❌ SALAH — src/audit.ts adalah dead code, tidak diimpor di mana pun
import { createAuditRecord } from "./audit.js";

// ✅ BENAR — createAuditRecord() dan appendAuditEvent() didefinisikan
// dan dipakai di src/runtime.ts (baris ~1535, ~1548)
await appendAuditEvent(
  ctx,
  createAuditRecord({
    kind: "registry.created",
    scope: "sikesra:registry",
    actor: actorFromContent(entity),
    summary: `Entitas ${entity.label} dibuat`,
    metadata: { entityId: entity.id, entityType: entity.entityType },
  }),
);
```

### HR-04: Storage Field Naming — `camelCase`, Bukan `snake_case`

Semua field di collection storage memakai `camelCase` (`entityType`, `villageCode`, `verificationStage`) — bukan `entity_type`/`village_code` seperti konvensi tabel SQL EmDash core. Jangan campur konvensi.

### HR-05: Jangan Asumsikan Tipe `fixtures.ts` Berlaku untuk Semua Collection

Hanya `registryEntities`, `supportingDocuments`, `verificationEvents` memakai tipe dari `src/fixtures.ts` secara langsung. Collection lain (`permissionCatalog`, `roleCatalog`, `abacAttributeCatalog`, `abacPolicyRules`, `auditEvents`) punya tipe lokal terpisah di `runtime.ts` dengan field tambahan. Lihat `04.DATABASE_SCHEMA.md` § 2.2 sebelum menulis kode yang mengasumsikan shape data.

### HR-06: Jangan Gunakan String Interpolasi di SQL

Tidak relevan langsung untuk plugin ini (tidak ada SQL ditulis manual), tetapi tetap berlaku untuk kode EmDash core lain yang mungkin disentuh secara tidak langsung.

### HR-07: KRITIS — Belum Ada Otorisasi Server-Side Nyata. Jangan Menambah Fitur Baru Tanpa Menyadari Ini

Hasil audit kode (Juni 2026): **tidak ada route mutasi (`*/save`, `verification/advance`, `verification/reject`, `import/promote`) yang memeriksa permission atau identitas terverifikasi sebelum menulis data.** Pengecekan level verifier di `verification/advance` membaca `verifierLevel`/`actor` langsung dari body request milik pemanggil — bisa dipalsukan oleh klien. `evaluateAbacDecision()`/`previewAccess()` hanya dipanggil oleh route preview/demo, tidak pernah menggerbang mutasi nyata.

**Jika tugas Anda menyentuh route mutasi apa pun**: jangan tambah fitur baru di atas fondasi yang tidak aman ini tanpa membaca `10.SECURITY_AND_PRIVACY_CHECKLIST.md` § temuan kritis dan mengonfirmasi scope tugas Anda (menambah fitur vs. memperbaiki otorisasi) dengan jelas di PR.

## 4. Invariant Inti (Terverifikasi)

```text
1. Plugin NATIVE: dir awcmsmicro-dev/packages/plugins/awcms-sikesra/
   npm: @ahliweb/awcms-sikesra
   plugin ID: awcms-sikesra (BUKAN awcms-micro-sikesra)

2. Storage: ctx.storage.<collectionName> — collections berindeks, BUKAN tabel SQL.
   Lihat AWCMS_SIKESRA_STORAGE di runtime.ts untuk daftar 18 collection.

3. Verification stages (9 nilai, urutan tetap):
   draft → submitted_village → verified_village →
   submitted_district → verified_district →
   submitted_sopd → verified_sopd →
   submitted_regency → active_verified

4. DUA set level yang TIDAK SAMA — jangan dicampur:
   VerificationUserLevel : desa_kelurahan < kecamatan < sopd < kabupaten < admin_sikesra
   VerificationLevel     : desa_kelurahan < kecamatan < sopd < kabupaten_admin < tampil
   (level "tampil" = stage active_verified, tidak mengizinkan verifier mana pun)

5. Sensitivity levels (akses semakin ketat):
   public_safe < internal < restricted < highly_restricted

6. Audit: WAJIB setiap mutasi, lewat appendAuditEvent()+createAuditRecord() di runtime.ts.
   kind format: "domain.action", e.g. "registry.created", "plugin.install"

7. Semua string user-facing via Lingui: t`...` atau <Trans>. Locale: en, id.

8. Admin UI: Kumo components ONLY. Tidak boleh custom button/input/dialog.

9. Tailwind: logical classes ONLY (ms-*, me-*, ps-*, pe-*, start-*, end-*)

10. Test: setiap bug harus punya failing test sebelum fix. Framework: vitest.

11. JANGAN tambah enforcement RBAC/ABAC baru tanpa membaca HR-07 dan
    10.SECURITY_AND_PRIVACY_CHECKLIST.md terlebih dahulu — banyak potongan
    kode (previewAccess, evaluateAbacDecision) TERLIHAT seperti enforcement
    nyata tapi sebenarnya simulasi read-only.
```
