## 3. Catatan Otorisasi (Kritis ŌĆö Baca Sebelum Memakai Endpoint Ini sebagai Contoh)

Grep penuh terhadap `ctx.user`, pengecekan permission, atau guard otorisasi di badan handler route: **tidak ditemukan satu pun** untuk route mutasi (`*/save`, `verification/advance`, `verification/reject`, `import/promote`). Satu-satunya gerbang kondisional yang ada adalah pengecekan level verifier di `verification/advance`/`verification/reject`:

```typescript
const allowedVerifierLevels = getAllowedVerifierLevels(item.currentLevel);
if (!allowedVerifierLevels.includes(verifierLevel)) { /* tolak */ }
```

tetapi `verifierLevel` dibaca langsung dari body request milik caller (`getString(routeCtx.input, "verifierLevel")`), dengan fallback ke `inferVerifierLevel(actor)` di mana `actor` juga dikirim caller ŌĆö **bukan diturunkan dari sesi/identitas yang terverifikasi**. Ini adalah pengecekan business-rule yang percaya klaim sendiri (self-asserted), bukan otorisasi nyata. `getRequestUserId()` membaca header `X-Sikesra-User-Id` yang tidak ditandatangani, hanya dipakai untuk filter tampilan di `verification/list`, tidak pernah sebagai gerbang mutasi.

`evaluateAbacDecision()`/`previewAccess()` hanya dipanggil dari route bertanda "preview"/"demo" (#27, #37, #38) ŌĆö **tidak pernah menggerbang mutasi nyata** di route manapun.

Lihat `10.SECURITY_AND_PRIVACY_CHECKLIST.md` untuk dampak dan rekomendasi hardening ŌĆö ini adalah gap yang harus ditangani sebelum plugin ini dipakai untuk data produksi sungguhan.
