# Monitoring, Evaluasi, dan KPI — SIKESRA

## 1. Kerangka Monitoring

Monitoring SIKESRA mencakup tiga lapis: kesehatan teknis sistem, adopsi/penggunaan oleh pengguna, dan kemajuan substansi (jumlah entitas terverifikasi per kategori dan wilayah).

## 2. KPI Teknis (System Performance)

| Metrik | Target Indikatif | Sumber |
| --- | --- | --- |
| Uptime endpoint health | ≥ 99% per bulan | `GET /health` |
| Waktu respons list registry (p95) | < 500ms | Query-count/perf snapshot |
| Error rate API plugin | < 1% request | Audit/error log |

### 2.1 Definisi Uptime

Persentase waktu `GET /_emdash/api/plugins/awcms-sikesra/health` merespons `{ data: { status: "ok" } }` dalam periode pengukuran.

## 3. KPI Operasional (Adopsi dan Penggunaan)

| Metrik | Definisi |
| --- | --- |
| Pengguna aktif per peran | Jumlah unik `actor` pada `auditEvents (collection)` per peran dalam 30 hari |
| Rata-rata waktu verifikasi per stage | Selisih `created_at` antar `verificationEvents (collection)` berurutan untuk entitas yang sama |
| Tingkat `needs_review` | Proporsi hasil `needs_review` dibanding total verifikasi — indikator kualitas input awal |

## 4. KPI Substansi (Registry Health)

### 4.1 Metrik Substansi

- Jumlah entitas per `entity_type` dan per stage verifikasi.
- Proporsi entitas yang mencapai `active_verified` dibanding total entitas yang dibuat.
- Cakupan wilayah: jumlah desa/kecamatan yang sudah memiliki minimal satu entitas terverifikasi.

### 4.2 Catatan MVP

Pada MVP, metrik ini dihitung dari query agregat langsung ke D1 (`registryEntities (collection)` GROUP BY), bukan dari pipeline data warehouse terpisah.

## 5. Metode Pengukuran

### 5.1 Dashboard Modul Reports

Modul **Reports** di admin UI menyediakan ringkasan KPI substansi secara real-time, dibatasi ABAC sesuai wilayah/scope pengguna yang melihat.

### 5.2 Review Berkala

Admin SIKESRA kabupaten melakukan review berkala (bulanan, indikatif) atas KPI operasional dan substansi untuk mengidentifikasi wilayah yang tertinggal dalam proses verifikasi.

## 6. Dashboard Monitoring

### 6.1 Komponen Dashboard

- Ringkasan jumlah entitas per tipe dan stage (kartu metrik).
- Daftar entitas yang "macet" (lama tidak berubah stage) — kandidat untuk tindak lanjut manual.
- Grafik tren submission per minggu (opsional MVP, dapat ditunda ke Phase 2 jika kompleksitas tinggi).

### 6.2 Hak Akses Dashboard

Dashboard Reports menggunakan permission `dashboardRead`/`auditRead` yang sama dengan modul Overview, difilter ABAC sesuai scope wilayah pengguna.

## 7. Evaluasi Berkala

### 7.1 Checklist Review Sprint

Setiap akhir sprint (lihat `09.SPRINT_EXECUTION_PLAN.md`), tim memeriksa: apakah exit criteria sprint terpenuhi, apakah ada KPI teknis yang merosot dibanding sprint sebelumnya.

### 7.2 Evaluasi Pasca-Rollout

Setelah rollout (lihat `14.TRAINING_AND_ROLLOUT_PLAN.md`), evaluasi KPI operasional dan substansi dilakukan secara berkala untuk memastikan adopsi berjalan sesuai harapan.

## 8. Eskalasi

### 8.1 Kondisi yang Memerlukan Eskalasi

- Uptime turun di bawah target selama lebih dari satu periode berturut-turut.
- Tingkat `needs_review` sangat tinggi pada satu wilayah tertentu — indikasi kebutuhan pelatihan ulang petugas desa di wilayah tersebut.
- Entitas yang macet di satu stage dalam waktu lama tanpa tindak lanjut — indikasi bottleneck verifikator.

### 8.2 Tindak Lanjut

Eskalasi ditangani secara internal oleh admin SIKESRA kabupaten; tindak lanjut teknis (bug, performa) dieskalasikan ke tim pengembang melalui isu GitHub.
